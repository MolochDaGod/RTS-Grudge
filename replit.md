# Survival & Combat Game

## Overview
A 3D survival and combat game built with React Three Fiber. Players explore a procedurally generated island terrain, gather resources, craft weapons, and fight waves of enemies. Features a richly populated medieval/fantasy world with NPC villagers, animals, and multi-type enemy waves. Inspired by Aurum (tigris-mt).

## Tech Stack
- **Frontend**: React + TypeScript + Three.js (via @react-three/fiber and @react-three/drei)
- **AI / Steering / NavMesh**: [Yuka](https://mugen87.github.io/yuka/) — wrapper at `client/src/game/ai/yuka/` (shared EntityManager via `useYukaTick`, `loadNavMesh`, `syncEntityToObject3D`)
- **Physics**: Rapier (Rust→WASM) via @react-three/rapier (RigidBody + collider components, HeightfieldCollider for terrain, CuboidCollider for buildings, BallCollider for player, interactionGroups-based collision filtering with 8 groups: TERRAIN/PLAYER/BUILDING/ENEMY/PROJECTILE/NPC/TRIGGER/RESOURCE, F3 debug wireframe toggle). **Fixed timestep** `1/60` with `interpolate={true}` for deterministic, smooth physics. Player RigidBody: dynamic, mass=5, linearDamping=0.5, enabledRotations=[false,false,false], ccd=true. Player BallCollider: friction=0.3, restitution=0.0. Terrain: RigidBody type="fixed" with HeightfieldCollider, friction=0.6. Buildings: RigidBody type="fixed" with CuboidCollider. **All lerps are delta-time based** using `1 - exp(-rate * dt)` for frame-rate independent smoothing (camera follow, slope modifier, idle deceleration, visual Y smoothing, model tilt). **Ground detection**: distance-based (`distToGround < 0.15 && vy < 2.0`) instead of velocity-only threshold. **MMO Camera System** (`Camera.tsx`): Two modes — **MMO** (pulled back, high-angle orbit, default distance 15) and **Action** (close over-shoulder, distance 5, shoulder offset 0.8). Right-click drag to orbit (yaw/pitch with pointer lock), scroll wheel zoom (smooth interpolated), V key toggles modes. `isCameraOrbiting()` export prevents RMB combat during camera drag. `triggerScreenShake()` for combat feedback. All settings from `useSettings`: cameraDistance (max zoom), cameraSensitivity (orbit speed), screenShake toggle. Exponential smoothing on follow (rate 8-12), orbit (rate 12-14), zoom (rate 10-12). Pitch clamped to prevent ground/sky extremes. **Spawn zone**: Player spawns at [0, 5, -5] in a cleared area near village center. Terrain flattened within 14 units of spawn. World objects excluded from spawn zone (12-unit radius).
- **Combat State Machine**: XState v5 (xstate + @xstate/react) — full combat state machine with 25+ states
- **Lightmap Baking** (`@react-three/lightmap` v0.0.8, opt-in): In-browser irradiance + AO baker. Wired through `client/src/game/lighting/BakedSceneWrapper.tsx` which (a) reads `?bakeLightmap=1` from the URL once at module load — no runtime toggle since baking happens at scene mount — (b) exposes `BakedSceneWrapper`, `LightmapIgnore`, `LightmapReadOnly`, and (c) re-types the upstream `Ignore`/`ReadOnly` components which ship with `React.FC` no-children typings (lib predates React 18's children change). Default settings tuned for ~30–50 m island scale: `ao: true`, `aoDistance: 4`, `texelsPerUnit: 4`, `lightMapSize: 1024`, `bounceMultiplier: 1`, `workPerFrame: 8` (4× upstream default → roughly halves wall-clock bake on a mid-range laptop). When the flag is absent the wrapper renders children as a passthrough fragment with zero overhead — `<Lightmap>` itself never mounts. Currently wired into `TutorialIslandScene` only: the static GLB world is the bake target, while `Sky`/`fog`/`VFXSystem`/`WaterSheets`/`UnderwaterVolume`/`PlacedBuildings`/`ShoreFoam`/`Player`/`PlayerBubbleTrail`/`Camera`/`TutorialIslandAnimals`/`TutorialIslandEnemies`/`TutorialIslandWaves`/`BreakApartChunks`/`LootDropsRenderer` are all wrapped in `<LightmapIgnore>`. Constraints (from the lib): only `MeshStandardMaterial`-derived meshes bake (Basic / Shader skipped automatically), skinned meshes are skipped automatically, geometry without `uv2` gets one auto-generated via potpack, only `directional/point/spot/ambient` lights and emissive materials contribute. To bake another scene: import `BakedSceneWrapper` + `LightmapIgnore` from the same file and follow the Tutorial Island pattern.

ModelEntry now includes optional `race`, `faction`, `gender`, `isPlayable`, `isWorgeForm`, `enemyScale` fields. **CharacterModelResolver** retains a fallback table that re-routes any stale legacy IDs (Knight/Wizard/Soldier/Viking/etc.) to thematically-matching new race assets — but all NPC, Enemy, ally, intro-cutscene, character-select, admin-panel, loading-screen, and animals path references have been rewritten to point directly at the new roster, so the resolver is now a backstop rather than a primary mapping. Presets apply model path, combat class, weapons, material colors, and body morphs. Hero name input, auto-save to localStorage, save/load custom builds. Interactive 3D preview (orbit/zoom/pan), combat class (melee/caster/archer), holdable weapons with proper grip transforms (computeWeaponTransform + buildWeaponGripData for skeleton-aware auto-positioning), per-material color customization (7 zones: skin/clothing/pants/hair/hat/armor/detail), AI model analysis, bone-based body morphing (18 sliders across 4 groups). **Weapon preview**: WeaponGizmo renders weapons at natural bone-local scale. ## Architecture

### Game Stores (`client/src/lib/stores/`)
- `useGame.tsx` - Game phase (menu/characterSelect/loading/playing/dead/paused/admin/gge), score, wave, day/night cycle. **Interaction Mode**: `interactionMode: "combat" | "build" | "harvest"` — Q key cycles modes, B key toggles build mode directly. `cycleInteractionMode()` rotates combat→build→harvest→combat. `setInteractionMode(mode)` for direct setting. Mode resets to "combat" on start/restart. `startLoading(config)` → loading screen → `finishLoading()` → playing. CharacterConfig includes: characterId, modelPath, name, scale, baseHeight, speedMultiplier, combatClass (melee/caster/archer), weaponRight, weaponLeft, materialColors (skin/clothing/pants/hair/hat/armor/detail — 7 keys), bodyMorph (18-param BodyMorphConfig), weaponOffset (WeaponOffsetConfig: rightPos/rightRot/rightScale/leftPos/leftRot/leftScale — position, rotation, and scale per hand), weaponModelRight/weaponModelLeft (optional FBX model IDs for 3D weapon models via WeaponModelLoader). `addXP()` delegates to `useCharacterStats.addExperience()` first (single authority), then mirrors level/xp/xpToNext back to useGame for legacy consumers
- `useGameConfig.tsx` - Full game configuration store with undo/redo history (50 states), export/import JSON. 11 config sections: TerrainConfig (worldSize, resolution, maxHeight, seed, noise params, water/snow levels), EnemyTypeConfig[] (per-enemy stats, behavior, spawning), WaveConfig (scaling, boss waves), NPCConfig[] (identity, position, dialogue, vendor), PhysicsConfig (gravity, speed, slope, collision), CombatConfig (damage, crit, defense), UIConfig (HUD, minimap, damage numbers), WorldConfig (day/night, lighting, fog, objects), BehaviorConfig (aggro, movement, idle), AnimationConfig (blend timing, root motion), StatsConfig (base stats, regen, leveling)
- `useSurvival.tsx` - Health, hunger, stamina, damage types (blade, impact, pierce, burn, poison, fall, starve). **Full stat-based defense pipeline**: uses `registerStatLookup()` pattern to receive 12 combat stats (defense, resistance, dodge, block, blockEffect, armor, damageReduction, fallDamage, critEvasion, ccResistance, bleedResist, evasion) without circular dependency. **Damage pipeline**: dodge check (dodge + evasion*0.5) → defense/armor mitigation (armor*0.5 added to physical def) → damageReduction% cap(50%) → bleedResist for blade damage → block check with computed blockEffect% cap(90%). **Fall damage**: applies fallDamage stat as % reduction cap(50%). **Hunger Status Tiers**: `getHungerStatus(hunger)` → well_fed (>80: +25% stamina regen), normal (30-80: baseline), hungry (<30: -30% stamina regen), starving (0: damage + -50% stamina regen). `regenStamina()` handles stamina regen only — health regen is exclusively via `useCharacterStats` healthRegen stat applied in Player.tsx healthRegenTimer.
- `useCharacterStats.ts` - **Grudge Warlords stats system**. 8 primary attributes (STR/VIT/END/INT/WIS/DEX/AGI/TAC) → 37 secondary stats via `computeSecondaryStats()`. 6 races (human/elf/dwarf/orc/barbarian/undead) with RACE_BONUSES (includes attribute bonuses applied in `getSecondaryStats`: Human +2TAC/+1WIS, Elf +2INT/+1AGI, Dwarf +2END/+1VIT, Orc +2STR/+1VIT, Barbarian +2AGI/+1DEX, Undead +1INT/+1END/+1TAC — matching grudge-studio.com reference). 4 hero classes: Warrior (STR+3/END+2/VIT+1), Mage (INT+3/WIS+2/TAC+1), Ranger (DEX+3/AGI+2/WIS+1), Worge (VIT+2/STR+2/AGI+2). 6 weapon masteries. 27 characters in `HERO_DEFINITIONS`. **CDN icons**: `CDN` base at `https://molochdagod.github.io/ObjectStore`, `ATTRIBUTE_ICON_BASE` for `icons/wcs/attributes/{attr}.png`, `ATTRIBUTE_ICON_ALTS` for alternate weapon/misc icons (Hammer, Book, Core, Dagger, Shield, Staff, Slash, Tablet). Race `iconUrl` fields link to CDN. `computeCombatDamage()` with dodge/crit/block rolls. **8 Tier Colors** (grudge-studio spec): Common=#8b7355, Uncommon=#a8a8a8, Rare=#4a9eff, Epic=#9d4dff, Legendary=#ff4d4d, Mythic=#ffaa00, Ancient=#d4a84b, Artifact=#f0d890. **Stats→Physics pipeline**: `movementSpeed` multiplies Player.tsx movement, `attackSpeed` scales action timers (1/atkSpeed), `drainHealth` provides life steal on hits, all 12 combat stats flow through `registerStatLookup` to `useSurvival.takeDamage`.
- `useStatsBridge.ts` - Performance bridge between character stats and survival. `getCachedPlayerStats()` caches derived stats by version hash (attributePointsSpent + level + skills sum). `getPlayerStatsForSync()` returns health/stamina for survival cap sync. `initStatsBridge()` subscribes to `useCharacterStats` for immediate health/stamina sync to `useSurvival` (no timer delay). `syncStatsToSurvivalNow()` can be called manually. Initialized in `main.tsx` at boot.
- `useInventory.tsx` - Items, crafting recipes (6 categories: tools/weapons/food/consumables/materials/armor, 30+ recipes), survival building recipes (5 categories: fire/structure/furniture/defense/storage, 22 pieces including campfire, foundations, walls, doorways, windows, roofs, stairs, workbench, storage boxes, fences, palisades, spike traps), slot selection (36 slots). Item types: weapon/food/material/tool/consumable/building/armor. Food items have healAmount + hungerRestore + staminaRestore. `useItem()` consumes food/consumables. `craftSurvivalBuilding()` deducts ingredients for building placement.
- `useSurvivalBuilding.ts` - Survival building placement system. PlacedSurvivalBuilding tracking (uid, recipeId, position, rotation, size, color, health). Ghost preview with terrain snapping. R to rotate, click to place, Esc to cancel. Event-driven placement via 'survival-build-place' CustomEvent from MainPanel Building tab.
- `useAudio.tsx` - Sound effects and background music
- `useHarvest.tsx` - Harvesting state: isHarvesting, harvestAnimation, progress/duration, startHarvest/tickHarvest/cancelHarvest
- `useBuildSystem.ts` - RTS building placement system. Resources (wood/stone/gold), **35+ building definitions** across 5 categories (defense/military/economy/housing/special), 2 ages (First/Second), up to 3 levels each. **New recruit buildings**: Farmer's Hut (auto-harvesting farmers), Warrior Hall (heavy melee), Ranger Lodge (patrol+harvest), Mage Tower (ranged fire, 2nd age), Wizard Sanctum (area buffs, 2nd age), Captain's Quarters (ally buff aura, 2nd age). **Ecosystem buildings**: Camp (spawns 2 farmers + berry/fiber/herb), Lumber Camp (wood+fiber), Herb Garden (herb+berry), Mining Outpost (stone+iron_ore), Crystal Grove (crystal+gold_ore). Farm I/II/III also spawn herb+berry+fiber. `SpawnResourceDef` type-safe with ResourceType union. `registerBuildingRemoveCallback()` pattern for cleanup on building destruction (avoids circular dependency with ResourceNode). BUILDING_REGISTRY with costs, sizes, ally spawning, spawnResources. PlacedBuilding tracking. Build mode (B key), ghost preview, grid snapping, R to rotate. Resource income from economy buildings every 10s.
- `useAllies.ts` - AI ally system. **10 ally types**: soldier, archer, knight, elite_archer, farmer, warrior, ranger, mage, wizard, captain. **Behavior states**: idle, patrol, combat, harvest, return_to_camp, follow, sleep, go_home. Farmers auto-harvest real ResourceNode instances via `getResourceNodesNear()` + `harvestNodeByIndex()` — path to node, harvest, deposit to inventory, return to camp, repeat. Rangers patrol + harvest. Mages/wizards use ranged projectiles (fireball/lightning). Captains provide damage buff aura (12-unit radius). Spawned from military/economy buildings. **Ally Command System**: `AllyCommand` type (follow/patrol/stay/attack_target), F1-F4 keyboard shortcuts, `setGlobalCommand()` updates all ally behaviors. **Per-ally personal command**: `personalCommand` + `personalTargetId` override the global squad order without touching squadmates (set via `setPersonalCommand(id, command, targetId?)`). **Identity & progression**: each ally has a `uuid` (crypto.randomUUID), human `name` (fantasy first-name table), `profession` (mapped from type), `level` 1-100, `xp`, `xpToNext` (curve `50 * level^1.6`); `awardXp()` cascades level-ups granting +5% maxHealth and +3% damage per level. **Day-night schedule**: non-combatant allies (`canHarvest` or low damage) walk to their `homePosition` at dusk and `isSleeping` until dawn; combat threats wake them and clear the sleep flag. **Telemetry**: `kills` and `resourcesGathered` counters update on damage/harvest. UI selection via `selectedAllyId` + `selectAlly()`.
- `AllyDetailPanel.tsx` - 540px modal opened by clicking an ally card in MainPanel. Cinzel/Crimson typography, gold theme. Tabs: Status (level/xp/health/profession/uuid + kills/resources), Orders (per-ally Follow/Patrol/Stay/Attack Target buttons that only affect the selected ally), Equipment (weapon/armor slots stub), Schedule (sleep/work hours + Set Home button). Uses `setPersonalCommand` and `setHomePosition`. Mounted in HUD.
- `useCombatLog.ts` - Real-time combat log Zustand store. `CombatLogEntry` with id, text, color, timestamp. `addEntry()` appends with 50-entry max. Used by Player.tsx (outgoing damage, kills, ally commands), Enemy.tsx (incoming damage), and HUD.tsx (scrolling display). Replaces static combat log messages.
- `useHarvest.tsx` - Player harvesting state: isHarvesting, harvestAnimation, progress/duration. Tick-based progress with callback on completion.
- `useTargeting.ts` - Tab-targeting system. Zustand store with targetId, setTarget, clearTarget, cycleTarget(enemies, playerPosition, maxRange). Cycles through nearby alive enemies sorted by distance. Tab key to cycle, Esc to clear (prioritized over pause).
- `useSettings.ts` - Player settings store. Graphics (quality/shadows/particles/postProcessing/viewDistance), Audio (master/music/sfx/ambient volumes, mute), Gameplay (damage numbers/health bars/minimap/FPS/camera distance/sensitivity/auto-attack/screen shake), Keybinds (display only, not yet remappable).
- `useCampaign.ts` - **Campaign mode store**. Tracks island progression, dungeon clearing, boss defeats, quest system. `CampaignIsland` with gridX/gridZ/seed/biome/difficulty/resources/wilderness flag. ### Progression System
- **XP/Leveling**: XP gained on kills (15 + 50% of base damage), XP threshold = 100 * level
- **Combo Counter**: Increments on successive hits, 2s decay timer, damage multiplier (1.0 + min(combo * 0.1, 1.0))
- **Critical Hits**: Base 5% chance, increases +2% per level (max 35%), 2x damage on crit
- **Skill Cooldowns**: Skills 1-5 have individual cooldowns (3-5s), HUD shows cooldown timers with fill overlay
- **Screen Shake**: Camera-based shake on crits (0.15s) and heavy hits (0.1s), non-destructive
- **Enemy Hit Feedback**: White flash on hit (0.12s), knockback on heavy hits (>10 damage), proportional to damage

### Unified Model Pipeline
All model loading routes through `AssetPipeline.importFromScene()` regardless of format. **Scale Pipeline**: `importFromScene` preserves the original GLTF root scale (no longer resets to 1,1,1), measures bounds via `computeSkinnedBounds()`, computes scale multiplier `targetHeight / measuredHeight`, applies via `multiplyScalar()` to preserve skeleton bind matrices. After scale change, `rebindSkeletons()` recalculates bone inverses and bind matrices. Post-scale verification re-measures and applies correction if >15% off target. `useCharacterModel` adds final emergency safety cap (MAX_ALLOWED_HEIGHT = max(3x target, 5.0)). **Character scale default is 1.0** (not 0.1), so `targetHeight = baseHeight * scale = 1.8`. **Animation Scale Track Safety**: `stripScaleTracks()` uses `lastIndexOf('.')` to correctly handle dotted object names (e.g. `Armature.001.scale`), strips ALL `.scale` tracks from ALL animations before retargeting. `retargetClipsToScene()` always strips scale tracks upfront. Track property parsing throughout uses `lastIndexOf('.')` not `indexOf('.')`. **BoundsUtils.ts**: `computeSkinnedBounds()` properly handles SkinnedMesh — samples actual skinned vertex positions (up to 2000 per mesh) using bone matrices, bind matrices, and skin weights for accurate bounds computation. Falls back to geometry bounding box for non-skinned meshes. Robust fallbacks to `setFromObject` and default 1.8m bounds. **Skeleton Rebinding**: `rebindSkeletons()` in AssetPipeline and `rebindAllSkeletons()` in useCharacterModel recalculate bone inverses + bind matrices after any scale change, handles both attached and detached bind modes.
- **Characters** (GLTF/GLB): `useCharacterModel` hook → `SkeletonUtils.clone` → `importFromScene()` → auto-retarget → `AnimationMixer`
- **World objects** (FBX/GLB): `FBXModel`/`GLBModel` in World.tsx → `importFromScene({ fixDarkMaterials: true })`
- **Weapons** (FBX): `WeaponModelLoader` → `FBXLoader` → `importFromScene({ textureAtlas })`
- **Dungeon decor** (FBX): DungeonScene.tsx → `importFromScene({ fixDarkMaterials: true })`

**Standard Rigging — Bone Name Normalization at Import Time**: All character skeletons are renamed to canonical Mixamo names (`mixamorigHips`, `mixamorigSpine`, etc.) during `importFromScene({ normalizeSkeleton: true })`. This makes every model's bones match the 231 animation GLBs directly — no runtime retargeting needed.

**Hierarchy-Aware Spine Detection**: Instead of relying solely on name aliases (which fails on reversed naming like `Spine02→Spine01→Spine`), the system walks the bone tree from hips toward head and assigns `spine/spine1/spine2` based on chain position. Handles all 4 skeleton types:
- KayKit (23 chars): `Body→Hips→Abdomen→Torso→[Chest]` → renamed to `mixamorigHips→mixamorigSpine→mixamorigSpine1→[mixamorigSpine2]`
- Mixamo (3 chars): Already canonical, no rename needed
- Generic (5 chars): `Hips→Spine02→Spine01→Spine` → correctly detected as bottom-to-top chain
- Missing bone fallbacks: `spine2→spine1`, `leftToe→leftFoot`, `rightToe→rightFoot`, `leftHand→leftLowerArm`

**Finger-Parent Hand Detection**: When no explicit hand/wrist bone exists (e.g., Anne.glb), detects finger children (Pinky/Index/Middle/Thumb) on the forearm bone and uses the forearm as the hand attachment point.

**Fallback Track Mapping**: For bones that don't physically exist on a model (e.g., spine2 on simple KayKit), animation tracks are redirected to the nearest parent bone via `BONE_FALLBACK_CHAIN`. This prevents animation data from being silently dropped.

### Save Game System (April 2026)
- **Server**: `server/saves.ts` — Neon-backed `game_saves` table (`(player_id, slot)` PK, JSONB `save_data`, `version` for optimistic locking, TIMESTAMPTZ created/updated). Works around three @neondatabase/serverless v1 HTTP-driver quirks: (1) `RETURNING` rows arrive empty so we follow up with `SELECT`, (2) `TIMESTAMPTZ` returns null over JSON so we cast `::text`, (3) zero-row queries throw `Cannot read properties of null (reading 'map')` so we catch that exact message and return `[]`. All user values flow through parameterised `sql.query(text, params)`.
- **Routes** (`server/savesRoutes.ts`, mounted only when `DATABASE_URL`/`GRUDGE_DATABASE_URL` is set):
  - `GET /api/saves/:playerId` → list summaries for all 8 slots
  - `GET /api/saves/:playerId/:slot` → full record incl. `save_data`
  - `PUT /api/saves/:playerId/:slot` → upsert; `expectedVersion` mismatch returns 409 with current row
  - `DELETE /api/saves/:playerId/:slot`
  - Validation: `^[A-Za-z0-9_-]{6,64}$` playerId, slot 0–7, 512 KB serialized body cap.
- **Client**:
  - `client/src/lib/save/playerId.ts` — random 32-char hex id stored in `localStorage.grudge.playerId`; active slot in `grudge.activeSlot` (default 0).
  - `client/src/lib/save/saveSync.ts` — explicit `snapshotGame()` / `restoreGame()` over `useCharacterStats`, `useEquipment`, `useInventory`, `useSurvival` only (Zustand `setState` preserves action functions; non-serialisable Maps/Sets in `useCampaign` are intentionally skipped). Plus `listSavesRemote/loadSlot/saveSlot/deleteSlot` REST helpers.
  - `client/src/lib/save/useAutoSave.tsx` — `<AutoSaveController />` mounted in `App.tsx`. On mount: pulls remote slot 0 and restores. Subscribes to the four stores; debounces changes by 4 s and PUTs while phase ∈ `playing`/`paused`. Final best-effort flush on `beforeunload`. Refuses to flush before the initial load resolves so default stores never clobber a real cloud save.
